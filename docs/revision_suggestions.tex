\section*{NeurIPS 2026 Review for Paper: COREY}

\subsection*{Summary}

This paper proposes \textbf{COREY}, an entropy-guided runtime chunk scheduling mechanism for selective scan kernels used in Mamba-style State Space Models (SSMs). The key idea is to use runtime activation entropy as a low-cost signal to dynamically select chunk sizes at kernel invocation time.

The system is evaluated across three tiers:

\begin{itemize}
\item \textbf{Tier 1}: Prototype cost-model simulations showing reductions in surrogate latency and DRAM traffic.
\item \textbf{Tier 2a}: Inline runtime entropy computation inserted into the Mamba inference pipeline, measuring scheduling overhead (reported as 8.3%).
\item \textbf{Tier 2b}: Independent kernel-level experiments demonstrating up to 4.41× latency reduction relative to static chunking.
\end{itemize}

However, the paper explicitly acknowledges that Tier 2a and Tier 2b are \textbf{not connected into a single end-to-end execution path}, leaving the main performance claim partially unverified.

Overall, the paper is positioned as a \textbf{Concept & Feasibility} contribution rather than a fully deployed runtime system.

\subsection*{Strengths}

\begin{enumerate}

\item \textbf{Clear and well-motivated systems problem}

The work targets a real bottleneck in selective scan pipelines, namely fragmentation-induced memory traffic overhead in GPU execution. The paper correctly highlights that theoretical $O(L)$ complexity does not guarantee hardware efficiency.

\item \textbf{Conceptually novel scheduling signal}

Using activation entropy as a runtime scheduling signal is an interesting idea that differs from:

\begin{itemize}
\item static fusion heuristics
\item offline profiling
\item arithmetic-intensity-only models
\end{itemize}

This introduces a dynamic, data-dependent optimization path that is theoretically meaningful.

\item \textbf{Structured experimental design}

The three-tier experimental design is logically organized:

\begin{itemize}
\item surrogate model (Tier 1)
\item runtime feasibility (Tier 2a)
\item kernel performance ceiling (Tier 2b)
\end{itemize}

This layered methodology helps isolate different components of the system.

\item \textbf{Transparent statement of limitations}

The authors clearly state several open gaps, including:

\begin{itemize}
\item absence of full end-to-end routing
\item limited workload diversity
\item missing architecture-level comparisons
\end{itemize}

This level of transparency is commendable.

\end{enumerate}

\subsection*{Weaknesses}

\subsubsection*{W1: Missing End-to-End Integration (Critical)}

The most important limitation is that runtime chunk decisions are \textbf{not actually routed into the live scan kernel}.

The paper explicitly states:

\begin{quote}
``Tier 2a and Tier 2b are separate experiments; connecting them remains the central open gap.''
\end{quote}

Without this integration:

\begin{itemize}
\item No true end-to-end latency improvement is measured.
\item The reported 4.41× speedup is not realized in real inference.
\item The system contribution remains incomplete.
\end{itemize}

This significantly weakens the empirical validity of the main claim.

This issue alone could lead to rejection in NeurIPS.

\subsubsection*{W2: Limited Real Workload Diversity}

All 80 tested prompts fall into a narrow entropy band:

\begin{itemize}
\item entropy range: 3.79–4.25 nats
\item all prompts mapped to the same chunk bucket
\end{itemize}

Therefore:

\begin{itemize}
\item dynamic adaptation is not demonstrated in real workloads
\item behavior collapses to static scheduling
\end{itemize}

This undermines the claim of dynamic runtime optimization.

\subsubsection*{W3: Insufficient Baseline Coverage}

Missing comparisons include:

\begin{itemize}
\item Mamba-2
\item RWKV-6
\item Transformer + FlashAttention-3
\end{itemize}

Without these:

\begin{itemize}
\item it is unclear whether COREY improves beyond strong baselines
\item the system advantage may be marginal
\end{itemize}

This limits the external validity of the results.

\subsubsection*{W4: Small Experimental Sample Size}

Key runtime experiments use:

\begin{itemize}
\item n = 5 repeats
\item short generation length
\end{itemize}

This weakens statistical reliability.

Standard NeurIPS systems papers typically require:

\begin{itemize}
\item n ≥ 20
\item longer sequences
\end{itemize}

\subsubsection*{W5: Theoretical Component Partially Detached}

The Hadamard-related theory is not experimentally validated.

Furthermore:

\begin{itemize}
\item empirical results contradict the theoretical entropy increase assumption
\item real checkpoints show entropy decrease
\end{itemize}

This reduces theoretical relevance.

\subsection*{Questions for Authors}

\begin{enumerate}

\item Can the authors provide a fully integrated Tier 2a + Tier 2b pipeline before camera-ready?

\item What is the end-to-end latency improvement after integration?

\item Does entropy vary significantly across real production workloads?

\item How does COREY compare against:

\begin{itemize}
\item FlashAttention-3
\item Mamba-2 SSD kernels
\end{itemize}

\item What is the memory overhead of histogram-based entropy estimation?

\end{enumerate}

\subsection*{Detailed Suggestions for Improvement}

\subsubsection*{Major Required Changes}

\begin{enumerate}

\item Implement full runtime integration.

Required experiment:

\begin{itemize}
\item route chunk selection into live selective_scan_fn
\item measure true end-to-end latency
\item report throughput improvement
\end{itemize}

This is the single most important revision.

\item Add cross-regime workload evaluation.

Suggested dataset mix:

\begin{itemize}
\item code generation
\item templated prompts
\item narrative text
\item long-context reasoning
\end{itemize}

Goal:

Demonstrate real chunk switching behavior.

\item Add modern systems baselines.

Minimum required:

\begin{itemize}
\item Mamba-2 SSD kernel
\item Transformer + FlashAttention-3
\end{itemize}

\item Increase runtime measurement reliability.

Required:

\begin{itemize}
\item ≥20 repeats
\item warm-up control
\item longer sequence lengths
\end{itemize}

\end{enumerate}

\subsubsection*{Medium Priority Changes}

\begin{enumerate}

\item Provide full computational complexity analysis.

Include:

\begin{itemize}
\item entropy computation cost
\item histogram update overhead
\item memory traffic estimates
\end{itemize}

\item Add ablation over entropy bin counts.

Suggested:

\begin{itemize}
\item K ∈ {32, 64, 128, 256}
\end{itemize}

\item Evaluate entropy stability under quantization.

\end{enumerate}

\subsubsection*{Minor Improvements}

\begin{enumerate}

\item Clarify notation consistency:

\begin{itemize}
\item distinguish raw entropy vs normalized entropy
\end{itemize}

\item Improve figure readability.

\item Move some appendix derivations into main text.

\end{enumerate}

\subsection*{Reproducibility Assessment}

\textbf{Score: 3 / 5}

Positive aspects:

\begin{itemize}
\item code repository provided
\item hyperparameters listed
\item hardware configuration described
\end{itemize}

Missing:

\begin{itemize}
\item full training logs
\item deterministic seeds
\item exact runtime pipeline configuration
\end{itemize}

\subsection*{Open Science Compliance}

The paper includes a code repository but does not fully specify:

\begin{itemize}
\item artifact versioning
\item evaluation scripts
\item dataset preprocessing steps
\end{itemize}

An explicit ``Open Science'' appendix is recommended.

\subsection*{Ethical and Societal Impact}

The broader impact discussion is reasonable and acknowledges:

\begin{itemize}
\item reduced energy cost benefits
\item possible misuse risks
\end{itemize}

No major concerns.

\subsection*{Overall Rating}

\textbf{Score: 5 / 10 (Borderline Reject)}

Justification:

The idea is promising and technically interesting, but the absence of full end-to-end validation significantly weakens the empirical contribution.

If integrated results are added, this score could improve substantially.

\subsection*{Confidence}

\textbf{4 / 5}

The evaluation is based on a detailed reading of:

\begin{itemize}
\item methodology
\item experimental design
\item limitation statements
\end{itemize}

The primary limitation (missing integration) is explicitly acknowledged in the manuscript.
