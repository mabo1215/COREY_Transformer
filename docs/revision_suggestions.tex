\section*{Paper Review: COREY — Entropy-Guided Runtime Chunk Scheduling for Selective Scan Kernels}

\subsection*{Summary}

This paper proposes \textbf{COREY}, a runtime scheduler for selective state space models (SSMs), particularly Mamba-1.x, that dynamically selects chunk sizes for selective scan kernels based on \textbf{activation entropy}. The method is positioned as a \textit{Concept \& Feasibility} contribution, aiming to demonstrate that entropy can serve as a lightweight runtime signal for kernel scheduling.

The empirical evaluation is organized into three tiers:
\begin{itemize}
    \item Tier-1: Python cost-model simulations showing reduced surrogate latency and DRAM traffic.
    \item Tier-2a: Real-checkpoint runtime hook measuring overhead (2.3\%–8.3\%).
    \item Tier-2b: Kernel-level benchmarks showing up to 4.41$\times$ speedup over static chunking.
\end{itemize}

The paper also includes a theoretical exploration of Hadamard-based entropy smoothing, although this component is explicitly marked as unvalidated and empirically falsified on real checkpoints.

\subsection*{Strengths}

\begin{itemize}
    \item \textbf{Novel perspective:} Introduces entropy as a runtime scheduling signal, which is conceptually simple and potentially generalizable.
    
    \item \textbf{Strong systems intuition:} Clearly identifies memory bandwidth and kernel fragmentation as bottlenecks in SSM inference.
    
    \item \textbf{Careful claim scoping:} The authors explicitly distinguish validated vs. unvalidated contributions, which aligns well with NeurIPS transparency expectations.
    
    \item \textbf{Multi-tier evaluation:} The separation into cost-model, kernel-level, and real-checkpoint experiments is well structured.
    
    \item \textbf{Reproducibility awareness:} Detailed appendix includes hardware, parameters, and experimental setup.
    
    \item \textbf{Negative result honesty:} The empirical falsification of Hadamard entropy gain is a strong scientific contribution.
\end{itemize}

\subsection*{Weaknesses}

\begin{itemize}

\item \textbf{Lack of real end-to-end speedup:}

The most critical issue is that COREY does not demonstrate meaningful end-to-end improvement on real workloads. The best reported result is near-parity:

\begin{itemize}
    \item Only $0.9905\times$ latency (≈0.95\% improvement) in routed H800 setup
\end{itemize}

This significantly weakens the practical impact.

\item \textbf{Entropy signal lacks variability:}

Across real workloads (LongBench, H800 stress tests):

\begin{itemize}
    \item Entropy is tightly concentrated (≈4.0 nats)
    \item All prompts map to the same chunk (256)
\end{itemize}

Thus, COREY effectively degenerates to a static scheduler in practice.

\item \textbf{No true system-level comparison:}

Missing comparisons against:

\begin{itemize}
    \item FlashAttention-3 (end-to-end)
    \item Mamba-2 optimized kernels
    \item Other runtime scheduling or autotuning systems
\end{itemize}

Current comparisons are limited to synthetic or kernel-level benchmarks.

\item \textbf{Prototype-heavy evaluation:}

A significant portion of results are based on:

\begin{itemize}
    \item Python cost model (not real GPU)
    \item Synthetic activations
\end{itemize}

This weakens the empirical credibility.

\item \textbf{Limited novelty in mechanism:}

The core idea reduces to:

\begin{quote}
Use a scalar statistic (entropy) to select chunk size
\end{quote}

This is relatively incremental compared to existing runtime autotuning methods.

\item \textbf{Hyperparameter sensitivity not fully explored:}

Key parameters:

\begin{itemize}
    \item $H_{ref}$
    \item histogram bins $K$
    \item thresholds $\tau$
\end{itemize}

are not systematically optimized or justified.

\item \textbf{No impact on model quality:}

The method does not evaluate:

\begin{itemize}
    \item perplexity changes
    \item downstream task accuracy
\end{itemize}

Thus, trade-offs between efficiency and quality remain unclear.

\end{itemize}

\subsection*{Technical Soundness}

\textbf{Rating: 6/10 (Moderate)}

\begin{itemize}
    \item The entropy formulation is mathematically valid.
    \item The scheduling heuristic is reasonable but not theoretically optimal.
    \item Theoretical claims (Hadamard) are explicitly invalidated on real data.
    \item Kernel-level results are convincing, but system-level conclusions are weak.
\end{itemize}

\subsection*{Empirical Evaluation}

\textbf{Rating: 5/10 (Weak)}

Major concerns:

\begin{itemize}
    \item Lack of diverse real workloads
    \item No ablation on entropy usefulness vs. simpler heuristics
    \item No comparison to learned or RL-based schedulers
    \item End-to-end gains are negligible
\end{itemize}

\subsection*{Clarity}

\textbf{Rating: 8/10 (Strong)}

\begin{itemize}
    \item Well-written and structured
    \item Clear separation of claims vs. limitations
    \item Extensive appendix improves transparency
\end{itemize}

\subsection*{Reproducibility}

\textbf{Rating: 8/10}

\begin{itemize}
    \item Code and data provided
    \item Detailed hardware and configuration descriptions
    \item Experimental protocols are explicit
\end{itemize}

\subsection*{Broader Impact \& Ethics}

\textbf{Rating: Adequate}

\begin{itemize}
    \item Discusses energy efficiency benefits
    \item Mentions potential misuse (e.g., surveillance scaling)
    \item No major ethical concerns
\end{itemize}

\subsection*{Overall Score}

\begin{itemize}
    \item \textbf{Score: 5 / 10}
    \item \textbf{Confidence: 4 / 5}
    \item \textbf{Recommendation: Weak Reject}
\end{itemize}

\subsection*{Main Reasons for Rejection}

\begin{enumerate}
    \item No convincing end-to-end performance gain
    \item Entropy signal lacks real-world variability
    \item Heavy reliance on synthetic/prototype evaluation
    \item Limited novelty relative to existing autotuning methods
\end{enumerate}

\subsection*{Suggestions for Improvement (Critical)}

\textbf{Must Fix for NeurIPS Acceptance:}

\begin{enumerate}

\item \textbf{Demonstrate real end-to-end speedup}

\begin{itemize}
    \item Integrate COREY into full inference pipeline
    \item Show $\geq 10\%$ latency or throughput gain
\end{itemize}

\item \textbf{Improve entropy signal effectiveness}

\begin{itemize}
    \item Explore alternative statistics:
    \begin{itemize}
        \item variance
        \item kurtosis
        \item token-level entropy
    \end{itemize}
\end{itemize}

\item \textbf{Expand workload diversity}

\begin{itemize}
    \item Include:
    \begin{itemize}
        \item code generation
        \item multimodal inputs
        \item long-context reasoning
    \end{itemize}
\end{itemize}

\item \textbf{Compare against stronger baselines}

\begin{itemize}
    \item Static optimal tuning (oracle)
    \item Learned schedulers
    \item Compiler-based fusion (e.g., XLA, nvFuser)
\end{itemize}

\item \textbf{Ablation on scheduler design}

\begin{itemize}
    \item Remove entropy → measure impact
    \item Replace entropy with random/constant policy
\end{itemize}

\item \textbf{Evaluate model quality impact}

\begin{itemize}
    \item Perplexity
    \item LongBench scores under COREY vs. baseline
\end{itemize}

\end{enumerate}

\subsection*{Minor Suggestions}

\begin{itemize}
    \item Clarify distinction between Tier-1 and Tier-2 claims earlier
    \item Reduce appendix redundancy
    \item Improve notation consistency (entropy normalization vs. raw nats)
\end{itemize}

\subsection*{Rebuttal Guidance (Very Important)}

To overturn this decision, authors must:

\begin{itemize}
    \item Provide \textbf{new experimental evidence} showing real gains
    \item Demonstrate entropy-driven switching on real workloads
    \item Justify why entropy is better than simpler heuristics
\end{itemize}

Without these, the paper is unlikely to be accepted.
