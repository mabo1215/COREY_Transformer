\section*{Summary}
%% ============================================================

This paper proposes COREY, a framework that combines
(i) entropy-guided fusion boundary selection and
(ii) fused Hadamard reparameterization for selective state space models (SSMs).
The core intuition is that activation entropy, estimated via binned histograms,
serves as a lightweight runtime signal to decide how aggressively to fuse
adjacent operators and how to size memory tiles.
Hadamard rotation is used to smooth heavy-tailed activations,
which in turn raises entropy and expands the safe operating region for deep fusion
and low-bit quantization.
Three theoretical results (entropy growth under doubly-stochastic mixing,
a shared-memory depth bound, and a variance-preservation/clipping argument)
are provided.
Experiments are conducted on a CPU-based Python prototype
(no GPU kernel is written or invoked),
and a minimal sanity check on a Mamba-1.4B checkpoint is reported.

%% ============================================================
\section*{Strengths}
%% ============================================================

\begin{enumerate}[leftmargin=*, label=\textbf{S\arabic*.}]

  \item \textbf{Clear conceptual link.}
    The paper cleanly unifies two previously separate threads
    (outlier suppression via Hadamard transforms and
    memory-aware operator fusion) by showing that entropy increase
    from rotation is simultaneously a quantization benefit and
    a systems scheduling signal.
    This is a genuinely useful framing.

  \item \textbf{Formal grounding.}
    Theorem~4.1 (entropy growth under doubly-stochastic mixing),
    Theorem~4.2 (shared-memory depth bound), and
    Theorem~4.3 (variance preservation with reduced coordinate extremes)
    are stated carefully and proved with correct applicability conditions.
    The authors are commendably honest that Theorem~4.1 is conditional,
    not an unconditional statement about every rotated distribution.

  \item \textbf{Transparent limitations.}
    Section~8 and Appendix~D explicitly enumerate what could not be run
    (PG19, GPTQ, AWQ, full Triton path, fast selective-scan kernels)
    and why.
    This transparency is welcome and should be preserved in revision.

  \item \textbf{Self-consistent prototype results.}
    Within the constraints of a CPU-level simulator,
    Tables~1--3 show internally consistent trends:
    larger gains at longer contexts, a clear latency--quality trade-off
    as $\tau$ increases, and monotone DRAM reduction.

\end{enumerate}

%% ============================================================
\section*{Weaknesses}
%% ============================================================

\begin{enumerate}[leftmargin=*, label=\textbf{W\arabic*.}]

  \item \textbf{[Critical] Experiments do not support the claimed contribution.}
    The paper's title, abstract, and introduction position COREY
    as a \emph{systems} contribution for GPU-resident SSM inference.
    However, every reported latency, throughput, and DRAM figure
    comes from ``Python-level timing of the simulated scheduler''
    on a CPU workstation (Section~5.1).
    No Triton or CUDA kernel is written, compiled, or benchmarked.
    The claimed metrics---latency in ms, tokens/s, DRAM bytes/token---are
    proxy values of a NumPy/PyTorch scheduler simulator,
    not real hardware measurements.
    A systems paper at NeurIPS must demonstrate real hardware gains;
    simulated proxies cannot substitute for them.
    \textcolor{weakcolor}{This is the most significant weakness
    and is likely to be the primary basis for rejection.}

  \item \textbf{[Critical] The ``quality drop'' metric is circular.}
    Section~5.4 defines ``proxy quality drop'' as a synthetic combination of
    a quantization sensitivity term, an over-fusion penalty,
    and a Hadamard bonus.
    The Hadamard bonus \emph{directly rewards} the proposed method's
    distinguishing feature.
    Consequently, the result that COREY achieves zero proxy quality drop
    (Table~4) is not an empirical finding but an arithmetic consequence
    of how the metric is defined.
    No perplexity or downstream task accuracy on real checkpoints
    is reported in the main results.

  \item \textbf{[Major] Checkpoint-level validation is insufficient.}
    The only real-model experiment (Appendix~D) runs Mamba-1.4B
    in FP32 on CPU (45.7~s latency, 2.80~tokens/s),
    which the authors themselves describe as unsuitable
    for the paper's systems claim.
    The GPU run achieves $\sim$21~tokens/s but still lacks
    the fast selective-scan path, so it cannot demonstrate
    the proposed fusion strategy.
    A fair comparison of COREY vs.\ baselines
    on any pretrained checkpoint is absent.

  \item \textbf{[Major] Entropy as a runtime signal is not validated.}
    The paper argues that per-region entropy is a useful scheduling signal,
    but it never shows that runtime entropy estimates are
    (a) stable across different inputs/batches,
    (b) cheap enough not to dominate kernel launch overhead, or
    (c) better than simpler proxies (e.g., channel variance, static profiles).
    Appendix~A.1 proposes an exponential moving average but
    provides no analysis of estimation variance or overhead.

  \item \textbf{[Major] Baseline comparison is weak.}
    ``Static Fusion'' is defined as a ``fixed fusion schedule
    without entropy signals'' (Section~5.1) but is not further specified.
    No existing SSM inference framework
    (e.g., FlashMamba, official Mamba-SSM fast path)
    is used as a baseline.
    Comparing against an unspecified internal baseline
    within a CPU simulator is not informative.

  \item \textbf{[Minor] The theoretical results have limited scope.}
    Theorem~4.2 ($F \leq M_{\text{shared}} / C_{\text{tile}}$)
    assumes a fixed, operator-uniform tile memory cost $C_{\text{tile}}$,
    which does not hold in practice
    (different operators have different register footprints).
    Theorem~4.3's ``qualitative bound'' is not quantified,
    making it difficult to assess whether the improvement is meaningful
    for specific bit widths.

  \item \textbf{[Minor] Writing and scope mismatch.}
    Abstract and introduction promise a contribution to
    GPU-efficient SSM inference, but Sections~5--6 reveal
    a prototype study.
    This mismatch will mislead readers.
    The paper should reframe itself as a proposal/analysis paper
    or deliver real kernel benchmarks.

  \item \textbf{[Minor] Reproducibility gaps.}
    The entropy estimation decay parameter $\lambda$,
    the weights $\alpha, \beta, \gamma$ in the fusion score (Eq.~4),
    and the tile-selection mapping are not given concrete values.
    Appendix~A.6 lists a checklist but does not fill it in.

\end{enumerate}

%% ============================================================
\section*{Questions for Authors}
%% ============================================================

\begin{enumerate}[leftmargin=*, label=\textbf{Q\arabic*.}]

  \item Can you provide real GPU kernel benchmarks
    (even on a single sequence length and a single model size)
    comparing COREY against the official Mamba-SSM fast path?
    If not, what prevents this, and why should the systems claims
    be believed?

  \item How is ``proxy quality drop'' calibrated against
    actual perplexity or downstream task accuracy?
    Have you measured any correlation between the synthetic proxy
    and real model quality metrics?

  \item What are the concrete values of $\alpha$, $\beta$, $\gamma$,
    and $\lambda$ used in the reported experiments?
    How sensitive are results to these hyperparameters?

  \item The doubly-stochastic condition in Theorem~4.1 is
    ``conditional.''
    Have you empirically verified that this condition holds
    (or approximately holds) for the activations encountered
    in Mamba-370M or Mamba-1.4B?
    If not, what evidence supports the applicability of
    Hadamard rotation to these specific models?

  \item How does the runtime cost of entropy estimation
    (histogram binning + EMA update) compare to the
    kernel launch and fusion overhead it is supposed to guide?
    Is there a regime where the estimation cost dominates?

\end{enumerate}

(Section~8, Appendix~D).
No claims about societal impact are made.

\vspace{6pt}
\noindent
The paper addresses an important and timely problem---efficient GPU inference
for selective SSMs---and the conceptual link between Hadamard smoothing,
entropy increase, and fusion safety is a useful insight.
However, the experimental evidence does not yet support the systems claim.
Latency and throughput figures are simulated on CPU,
the quality metric is defined to favor the proposed method,
and no comparison against existing SSM inference backends
on real hardware is provided.
The paper should be substantially revised to either
(a) deliver real Triton/CUDA kernel benchmarks on pretrained SSMs,
or (b) reframe itself as a theoretical proposal with honest simulation results,
removing the hardware efficiency claims it cannot currently substantiate.

%% ============================================================
\section*{Detailed Suggestions for Revision}
%% ============================================================

\begin{enumerate}[leftmargin=*, label=\textbf{R\arabic*.}]

  \item \textbf{Replace simulated metrics with real kernel benchmarks.}
    At minimum, implement the fused Hadamard + selective-scan kernel
    in Triton for a single operator type (e.g., the SSM projection block)
    and report wall-clock latency and DRAM throughput
    on one GPU (e.g., A100 or RTX 3090) for one model size.
    Even a single sequence length would substantially strengthen the paper.

  \item \textbf{Replace the proxy quality metric with perplexity.}
    Run at least one pretrained Mamba checkpoint
    under FP16 and W8A8 quantization and report WikiText-103 perplexity
    for native inference, static fusion, and COREY.
    If quantization tooling is unavailable,
    restrict claims to FP16 and remove quantization-related assertions.

  \item \textbf{Specify all hyperparameters.}
    Report $\alpha$, $\beta$, $\gamma$, $\lambda$, $\tau_0$,
    the number of histogram bins $K$,
    and the tile size mapping in one consolidated table.

  \item \textbf{Fix the framing.}
    If real kernels cannot be provided by submission time,
    reframe as a ``framework proposal and prototype study''
    and remove language implying demonstrated hardware efficiency.
    The abstract currently reads as if GPU gains have been measured,
    which is inaccurate.

  \item \textbf{Quantify Theorem~4.3.}
    Replace the qualitative overflow-probability bound
    with a quantitative version parameterized by the tail index
    and the clipping threshold $k$,
    so readers can assess its practical significance
    for 4-bit and 8-bit quantization.

  \item \textbf{Add a cost analysis of entropy estimation.}
    Provide a theoretical or empirical estimate of the overhead of
    histogram computation and EMA updates relative to kernel execution time,
    to justify that the signal is ``lightweight'' as claimed.

  \item \textbf{Compare against a named baseline.}
    Use the official \texttt{mamba-ssm} Python path
    (even without the fast selective-scan kernel, as that is currently
    the accessible baseline) as the ``No Fusion'' reference,
    and document exactly what ``Static Fusion'' does.

\end{enumerate}
