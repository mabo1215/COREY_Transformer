\section*{Summary}
%==============================================================================

COREY is a runtime chunk-size scheduler for the Mamba-1.x
\texttt{selective\_scan\_fn} Triton kernel, using Shannon entropy estimated
from fixed-width histograms to map activation entropy to a chunk size via a
log-scale rule. The paper is positioned as a \textbf{Concept \& Feasibility}
contribution organized in three evidence tiers:
\begin{itemize}
  \item \textbf{Tier~1}: Python cost-model prototype establishing entropy-guided
  grouping reduces surrogate latency and DRAM traffic.
  \item \textbf{Tier~2a}: Active inline hook measuring $8.3\%$ end-to-end
  overhead when all 48 Mamba-370M layers are instrumented; chunk decisions are
  computed inline but not yet routed into the scan kernel.
  \item \textbf{Tier~2b}: Direct Triton selective-scan kernel timing showing
  $3.24\times$ (legacy $H_{\text{ref}}{=}8.0$, chunk$=256$) and
  $\mathbf{4.41\times}$ (calibrated $H_{\text{ref}}{=}\log K$, chunk$=512$,
  RTX~3070) speedup over Static-64; the calibrated result matches the
  Static-512 oracle exactly.
\end{itemize}

All Cycle~6 required patches (A and B) are confirmed applied in the current
manuscript: the Conclusion correctly headlines the $4.41\times$ calibrated
result (not the legacy $3.24\times$), and the RTX~3090 COREY (legacy) row
carries Speedup-B$=0.73\times$. The paper reads as a \textbf{solid Accept}
(7/10) at the Concept \& Feasibility tier.

This Cycle~7 review identifies five remaining presentation-level issues---none
requiring new experiments---that, if addressed, would produce a fully
consistent, submission-ready manuscript.

\section{Weaknesses}
%==============================================================================

\subsection{W1 (medium) --- ``chunk 288'' inconsistency with the discrete chunk
formula.}

Section~6.1 (Signal-to-decision-to-latency summary) states: ``the runtime hook
records 4.18~nats and recommends chunk~288.'' Section~6.2 describes the COREY
formula as:
\[
  C_{\text{corey}} = \text{clip}\!\left(2^{\lfloor\log_2(C_{\min} + r(C_{\max}
  - C_{\min}))\rfloor},\; C_{\min},\; C_{\max}\right)
\]
with $C_{\min}{=}32$, $C_{\max}{=}512$, and $r=\min(H/H_{\text{ref}},1)$.
For $H{=}4.18$\,nats and $H_{\text{ref}}{=}5.55$\,nats, $r=0.753$.
$C_{\min} + r(C_{\max}-C_{\min}) = 32 + 0.753\times480 = 393.4$.
$2^{\lfloor\log_2(393.4)\rfloor} = 2^8 = 256$.

The formula yields \textbf{chunk$=256$}, not chunk$=288$.
Chunk~288 ($=2^8+32$) is not a power of 2 and is not representable as an
output of the described discrete formula. A similar inconsistency appears later
in Section~6.1, where the real-checkpoint distribution text states ``the
continuous runtime recommendation is 288 for all 80 prompts.'' This implies
the hook runs a \emph{continuous} (non-rounded) variant of the formula,
distinct from the power-of-2 formula shown in Section~6.2.

\textbf{Required resolution (Patch~A):} Either (a) clarify explicitly in
Section~6.1 that ``288'' is the \emph{continuous} (non-rounded) output of the
formula before power-of-2 discretization and that the discretized value used
by the benchmark is chunk$=256$, or (b) replace all occurrences of ``chunk 288''
with ``chunk 256 (continuous formula output: 393, discretized to nearest
power-of-2 bin: 256)'' if the benchmark consistently uses the rounded value.
The first occurrence in Section~6.1 and the repeated statement in the
real-checkpoint paragraph should both be updated.

\subsection{W2 (low) --- RTX~3090 Static-64 row missing Speedup-B.}

The RTX~3070 block of Table~1 (\texttt{tab:real-gpu-three-policy}) provides
Speedup-B$=0.21\times$ for the Static-64 row. The RTX~3090 block shows
``---'' for that same row's Speedup-B. The value is straightforwardly
computable: Static-512 oracle latency for RTX~3090 is $0.022$\,ms;
Static-64 latency is $0.078$\,ms; therefore Speedup-B$=0.022/0.078
\approx0.28\times$.

The table caption asserts ``Speedup-B is now populated for all non-TPU/T4
rows on both RTX~3070 and RTX~3090,'' but this is only true for the COREY
rows---the RTX~3090 Static-64 row still shows ``---''. This asymmetry is
minor but contradicts the caption.

\textbf{Required resolution (Patch~B):} Add Speedup-B$\approx0.28\times$ to the
RTX~3090 Static-64 row. Update the caption sentence to enumerate all populated
rows (Static-64 and COREY) consistently across both hardware platforms.

\subsection{W3 (low) --- Contributions (2) misleadingly attributes
3.24$\times$ to ``the chunk-size granularity selected by COREY''.}

Contributions paragraph (2) states: ``(Tier~2b) a separate Triton kernel-level
scan benchmark showing $3.24\times$ speedup at the chunk-size granularity
selected by COREY (calibrated setting: $4.41\times$, \ldots).''

The phrase ``chunk-size granularity selected by COREY'' without qualification
now reads as the calibrated default, which is chunk$=512$ and $4.41\times$.
The $3.24\times$ figure belongs to the \emph{legacy} $H_{\text{ref}}{=}8.0$
setting (chunk$=256$), not to the paper's current default. A reader who skims
only Contributions will conclude that the paper's primary COREY result is
$3.24\times$---which contradicts the abstract, the table, and the Conclusion.

\textbf{Required resolution (Patch~C):} Rewrite the Tier-2b bullet to lead
with the calibrated result and identify the legacy result explicitly.

\subsection{W4 (low) --- Table caption: ``COREY (default) and COREY
(calibrated) are separate runs'' is misleading.}

The Table~1 caption states: ``COREY (default) and COREY (calibrated) are
separate runs; the calibrated setting is the paper default.''

In the current table, there is only one COREY row per hardware block labeled
``COREY (default, $H_{\text{ref}}{=}\log K$).'' There is no row explicitly
labeled ``COREY (calibrated)'' distinct from ``COREY (default).'' If both
names refer to the same configuration ($H_{\text{ref}}{=}\log K$, chunk$=512$),
the sentence ``COREY (default) and COREY (calibrated) are separate runs'' is
confusing: it implies two different experimental configurations that are not
visible in the table.

\textbf{Required resolution (Patch~D):} Remove or rewrite the
``COREY (default) and COREY (calibrated) are separate runs'' clause. State
instead that the COREY (default) row was measured in a dedicated 30-repeat
timing run, if that is the relevant distinction.

\subsection{W5 (very low) --- NeurIPS 2026 Checklist is commented out.}

The paper includes a commented-out NeurIPS~2026 mandatory checklist
(end of main .tex file). The NeurIPS submission system requires the checklist
to be submitted as part of the main PDF. The answers already present in the
comments are appropriate and complete.

\textbf{Required resolution (Patch~E):} Uncomment the checklist section before
final submission. The content of the answers does not need to change.
