\section*{Summary}
%==============================================================================

COREY is a runtime chunk-size auto-tuner for the Mamba-1.x
\texttt{selective\_scan\_fn} Triton kernel, using activation entropy as the
scheduling signal.  Evidence spans three tiers: Tier~1 (Python cost-model
prototype), Tier~2a (real-checkpoint inline hook, $8.3\%$ end-to-end overhead,
$n{=}5$), and Tier~2b (direct Triton kernel timing, calibrated $4.41\times$
over Static-64 on RTX~3070, $n{=}30$).

All Cycle~9 required revisions (Patches~A through~C) are confirmed applied in
the current manuscript:
\begin{itemize}
  \item Patch~A: $\lfloor\log_2(\cdot)\rceil \to \lfloor\log_2(\cdot)\rfloor$
    bracket fix in \texttt{appendix.tex} Section~A.14 ---
    \textbf{applied}.
  \item Patch~B: Appendix~A.1 cross-reference updated from stale
    \texttt{tab:w1\_chunked\_scan} to \texttt{tab:real-gpu-three-policy}; RTX
    3070 latency text updated from $0.748$\,ms to $0.013$\,ms with
    methodology note; dual-TPU hardware explanation added ---
    \textbf{applied}.
  \item Patch~C: Legacy $H_{\text{ref}}{=}8.0$ warning sentences appended to
    \texttt{tab:chunk\_sweep} and \texttt{tab:real\_checkpoint\_entropy}
    captions --- \textbf{applied}.
\end{itemize}

This Cycle~10 review identifies three remaining issues: (1)~the
\emph{Triton Integration Notes} in Appendix~A.2.5 contains a tile-size mapping
formula $T(e)$ that implicitly uses the legacy $H_{\text{ref}}{=}8.0$
normalizer, inconsistent with the paper default $H_{\text{ref}}{=}\log K$;
(2)~\texttt{tab:chunk\_sweep} (Appendix~A.12) bolds the legacy
\texttt{corey-256} row rather than a default-calibrated row, creating visual
tension with the main-text Table~1 which bolds the default \texttt{COREY} row
at chunk~$=512$; (3)~\texttt{tab:hook\_micro} (Appendix~A.16) shows negative
latency deltas for all three passive-hook rows without an explicit explanation
that the direction of the delta is noise-dominated at $n{=}1$.  All three
issues are pure text edits; no new experiments are required.

\section{Weaknesses}
%==============================================================================

%------------------------------------------------------------------------------
\subsection{W1 (low) --- $T(e)$ formula in Triton Integration Notes uses
legacy $H_{\text{ref}}{=}8.0$ implicitly}

Appendix~A.2.5 (Triton Integration Notes, \texttt{sec:appendix\_systems})
contains the tile-size mapping:
\[
  T(e) = 32 \cdot \operatorname{round}\!\left(
    \frac{64 + \tfrac{\min(e,8)}{8}\cdot(512-64)}{32}
  \right),
\]
where $\min(e,8)$ caps the entropy at $8.0$\,nats and divides by $8.0$---
exactly the legacy $H_{\text{ref}}{=}8.0$ normalizer.  Under the paper
default $H_{\text{ref}}{=}\log K{=}5.55$\,nats, the correct formula is
$\min(e,\log K)/\log K$.  The section is already labeled as
``\emph{prospective design target; not implemented or measured in this
submission}'', but the formula inconsistency would confuse a reader who
attempts to reproduce the described hook from this description.

\textbf{Required resolution (Patch~A):}  In \texttt{appendix.tex},
Appendix~A.2.5, add one sentence after the $T(e)$ formula noting that the
displayed formula uses the legacy $H_{\text{ref}}{=}8.0$ normalizer for
historical continuity; under the paper default $H_{\text{ref}}{=}\log K$, the
saturating term becomes $\min(e,\log K)/\log K$ and selects chunk$=512$
for standard-normal inputs ($H{=}4.60$\,nats, $r{=}0.83$).  Pure text
addition; no formula rewrite required.

%------------------------------------------------------------------------------
\subsection{W2 (low) --- \texttt{tab:chunk\_sweep} bolds the legacy
\texttt{corey-256} row, inconsistent with main-text Table~1}

The main-text Table~1 (\texttt{tab:real-gpu-three-policy}) bolds the
\textbf{COREY (default, $H_{\text{ref}}{=}\log K$)} row at chunk~$=512$,
$4.41\times$ speedup.  Appendix Table~A.12 (\texttt{tab:chunk\_sweep}) bolds
the \textbf{\texttt{corey-256}} row, which was measured under the legacy
$H_{\text{ref}}{=}8.0$ setting (chunk~$=256$, $2.87\times$ speedup).  A reader
who encounters Table~A.12 before Table~1---or who scans the appendix in
isolation---will infer that the primary COREY result is chunk~$=256$, directly
contradicting the main-text headline.

The table caption already contains a note about the legacy setting (added in
Cycle~9 Patch~C), but the visual hierarchy (bold row) still foregrounds the
legacy result.

\textbf{Required resolution (Patch~B):}  Add a new row
``\texttt{corey-512 (default, $H_{\text{ref}}{=}\log K$)}'' to
\texttt{tab:chunk\_sweep} immediately above \texttt{static-512}, with
chunk~$=512$, kernel~calls~$=8$, and latency values identical to those of
\texttt{static-512} (since COREY selects chunk~$=512$ on this workload and
both invoke the same kernel configuration).  Bold this new row as the default.
Change the \texttt{corey-256} bold to non-bold and label it
``\texttt{corey-256 (legacy, $H_{\text{ref}}{=}8.0$)}''.  Update the speedup
column for the new default row to $4.41\times$.  No new GPU runs are required
because \texttt{corey-512} and \texttt{static-512} differ only in \emph{how}
chunk~$=512$ is chosen, not in the kernel invocation itself.

%------------------------------------------------------------------------------
\subsection{W3 (low) --- \texttt{tab:hook\_micro} negative latency deltas
unexplained}

Appendix~A.16 (\texttt{tab:hook\_micro}) reports passive-hook feasibility
measurements.  All three rows show negative latency deltas:
$-3.63\%$, $-4.02\%$, and $-1.30\%$.  The table caption states
``Latency deltas are within measurement noise at these sample counts and
must not be interpreted as stable speedup estimates.'' However, it does not
explain \emph{why} a passive hook (which adds code to the forward path) could
produce a negative delta.  A reviewer encountering this table will reasonably
ask: is the hook somehow faster than no-hook, or is the experiment
mis-configured?

The explanation is straightforward: at $n{=}1$ with zero warmup, GPU cold-start
and process-scheduling variability can dominate the measured delta; the passive
hook has negligible GPU arithmetic so the apparent ``speedup'' reflects random
OS-level noise rather than any real acceleration.

\textbf{Required resolution (Patch~C):}  Append one sentence to the
\texttt{tab:hook\_micro} caption clarifying the physical interpretation:
``Negative delta values reflect GPU cold-start and OS scheduling noise at
$n{=}1$, zero-warmup; since the passive hook adds no GPU arithmetic, any
apparent latency \emph{reduction} is a noise artifact and should not be
interpreted as hook-induced speedup.''  Pure text addition.

\section{Minor Issues}
%==============================================================================

\begin{enumerate}[leftmargin=2em,label=m\arabic*.]
  \item \textbf{RTX~3070 comparison row in \texttt{tab:tpu\_corey\_benchmark}.}
    The table row ``RTX~3070 (static-512, Triton): $0.748\pm0.037$\,ms''
    cites the chunk-sweep-harness value (including Python dispatch), not the
    isolated Triton-kernel timing ($0.013\pm0.006$\,ms, main-text Table~1).
    The paragraph text in Appendix~A.1 now explains this distinction
    (Cycle~9 Patch~B), but the table row itself has no annotation.  Adding a
    ``$\dagger$'' footnote to that row---``$^\dagger$From the chunk-sweep
    harness (Table~\ref{tab:chunk\_sweep}); isolated Triton-kernel timing is
    $0.013\pm0.006$\,ms (Table~\ref{tab:real-gpu-three-policy})''---would make
    the table self-contained without changing the text.

  \item \textbf{Mamba-2.8B rows in \texttt{tab:policy\_compare\_n5}.}
    Both \texttt{policy\_static} and \texttt{policy\_corey} Mamba-2.8B rows
    remain pending (awaiting $n{\ge}20$ rerun to resolve the $n{=}5$ WT103 PPL
    anomaly).  This is correctly documented in the caption; no text change is
    needed.  This item is noted for completeness as a known open item.

  \item \textbf{Data-parallel speedup in Conclusion.}
    The Conclusion paragraph references data-parallel sharding speedup
    ($2.11\times$ on 2~GPUs, $3.45\times$ on 4~GPUs) with an explicit
    parenthetical that this is ``sample-level parallelism, independent of
    COREY's chunk scheduler.''  The result is not a COREY contribution but is
    included as supporting evidence for the checkpoint harness.  Replacing
    the numerical mention with a cross-reference to Section~\ref{sec:ckpt_status}
    would tighten the Conclusion, but this is cosmetic and not required for
    acceptance.
\end{enumerate}

\section{Questions for Authors}
%==============================================================================

\begin{enumerate}[leftmargin=2em,label=Q\arabic*.]
  \item \textbf{Integration timeline.}  Section~7 (Open experimental gaps)
    describes the planned end-to-end integration as modifying
    \texttt{MambaMixer.cuda\_kernels\_forward} to pass the entropy-selected
    chunk directly to \texttt{selective\_scan\_fn}.  This appears to be a
    small code change on top of the already-instrumented Tier-2a hook.  Can
    the authors confirm whether this integration is feasible before the
    camera-ready deadline, and if so, whether the resulting $n{\ge}5$
    end-to-end latency would appear in the paper?  If not, confirming the
    concrete blocker (e.g., API incompatibility in \texttt{transformers} 5.x)
    would strengthen the limitations statement.

  \item \textbf{$T(e)$ formula deployment target.}  The $T(e)$ formula in
    Appendix~A.2.5 uses $\min(e,8)/8$, which is the legacy normalizer.  Is
    this formula the one that would be used in the eventual Triton deployment,
    or is it a historical artifact that should be updated to
    $\min(e,\log K)/\log K$?  A one-sentence clarification (Patch~A) would
    resolve the ambiguity.
\end{enumerate}
