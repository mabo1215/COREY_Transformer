\section*{1. Summary}

This paper presents \textbf{COREY}, a prototype runtime scheduling framework 
designed to alleviate memory-bandwidth bottlenecks in 
Mamba-1.x Selective State Space Models (SSMs).

The central idea is to use \textbf{activation entropy} 
as a lightweight runtime signal to dynamically determine:

\begin{itemize}
\item chunk size selection
\item kernel fusion boundaries
\item memory-aware scheduling decisions
\end{itemize}

Entropy is estimated using fixed-width histograms, 
and decisions are made through a hardware-aware scoring mechanism.

The framework is evaluated in two stages:

\begin{itemize}
\item Tier-1: deterministic Python-based analytic cost model
\item Tier-2: GPU feasibility experiments on Mamba-370M and 1.4B checkpoints
\end{itemize}

Additionally, the paper proposes a theoretical analysis of 
Hadamard reparameterization and its effect on entropy behavior.

Overall, the paper explores an interesting intersection 
between information theory and runtime kernel scheduling, 
but the implementation remains at the prototype level, 
and several theoretical and empirical gaps limit the strength 
of the conclusions.

\section*{2. Strengths}

\begin{enumerate}

\item \textbf{Novel Runtime Signal Design}

Using activation entropy as a prompt-dependent scheduling signal 
is an original idea compared with traditional static fusion 
or offline profiling-based tuning.  
The formulation is conceptually interesting and 
could inspire future runtime-adaptive systems.

\item \textbf{Hardware-Aware Modeling}

The scheduling mechanism explicitly considers:

\begin{itemize}
\item register pressure
\item shared memory limits
\item occupancy constraints
\end{itemize}

This grounding in real GPU resource models 
strengthens the credibility of the design.

\item \textbf{Transparent Scope of Claims}

The authors clearly state that:

\begin{itemize}
\item full fused-kernel deployment is not yet implemented
\item end-to-end perplexity improvements are not claimed
\end{itemize}

Such clarity is appreciated and improves interpretability.

\item \textbf{Proof-of-Concept GPU Results}

Kernel-level latency measurements 
on RTX-class GPUs demonstrate measurable gains 
under optimized chunk settings, 
indicating the feasibility of the scheduling mechanism.

\item \textbf{Formal Theoretical Framing}

The paper includes multiple theorems 
relating entropy properties to system behavior, 
which is uncommon in systems-oriented ML work 
and adds intellectual depth.

\end{enumerate}

\section*{3. Major Weaknesses}

\subsection*{W1. Passive Hook (Missing Closed-Loop Execution)}

The most critical limitation is that 
the entropy scheduler remains \textbf{passive}.

Currently:

\begin{itemize}
\item entropy is computed
\item chunk recommendations are generated
\item but recommendations are \textbf{not actively applied}
to the runtime execution kernel
\end{itemize}

This prevents measurement of:

\begin{itemize}
\item end-to-end latency
\item true system-level benefit
\item real runtime overhead
\end{itemize}

Without closed-loop integration, 
the work remains a feasibility prototype 
rather than a deployable runtime system.

This is a major limitation 
for a systems-oriented NeurIPS submission.

\vspace{6pt}

\subsection*{W2. Limited Workload Diversity}

The real workload evaluation 
uses approximately 80 prompts 
derived from a LongBench subset.

Observed issue:

\begin{itemize}
\item entropy distribution is highly concentrated
\item scheduler mostly selects a single chunk bucket
\item dynamic behavior rarely occurs
\end{itemize}

This suggests the method behaves more like:

\begin{center}
\textbf{a static auto-tuner}
\end{center}

rather than a true dynamic controller.

The generalization capability 
remains insufficiently demonstrated.

\vspace{6pt}

\subsection*{W3. Theory–Practice Mismatch (Hadamard Behavior)}

The paper proves theoretical entropy growth 
under Hadamard rotation.

However, empirical results on real Mamba activations show:

\begin{center}
\textbf{entropy often decreases}
\end{center}

This contradiction weakens the central theoretical narrative.

The authors acknowledge this mismatch, 
but the explanation remains incomplete.

A theory validated only under synthetic heavy-tailed distributions 
limits real-world relevance.

\vspace{6pt}

\subsection*{W4. Insufficient Strong Baselines}

Comparisons with state-of-the-art systems are limited.

Missing baselines include:

\begin{itemize}
\item Mamba-2 / SSD
\item FlashAttention-3 style Transformer kernels
\item Profile-based static tuning frameworks
\end{itemize}

Without these, 
it is difficult to judge 
whether entropy-based scheduling 
provides unique advantages.

\vspace{6pt}

\subsection*{W5. Weak Demonstration of True Dynamic Adaptation}

Entropy values in real workloads 
show limited variance.

As a result:

\begin{itemize}
\item bucket switching rarely occurs
\item adaptive capability is not convincingly demonstrated
\end{itemize}

This challenges the claim 
that the system is ``prompt-responsive.''

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section*{4. Minor Weaknesses}

\begin{enumerate}

\item Some theoretical sections 
are lengthy relative to empirical validation.

\item Notation related to entropy thresholds 
(e.g., $H_{ref}$ selection) 
is introduced before practical motivation 
is fully established.

\item Reproducibility details 
(e.g., configuration scripts and profiling parameters) 
should be more explicitly documented.

\end{enumerate}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section*{5. Questions for Authors}

\begin{enumerate}

\item What technical barriers currently prevent 
forwarding entropy-derived chunk sizes 
into the actual execution kernel?

\item Can the authors provide evidence 
of real workloads where entropy variation 
triggers multiple bucket switches?

\item Why do real Mamba activations 
exhibit entropy contraction under Hadamard rotation?

\item How sensitive is the scheduler 
to histogram resolution ($K$)?

\item What is the expected runtime overhead 
of a fully active scheduler?

\end{enumerate}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\section*{6. Required Revisions (Mandatory)}

\begin{enumerate}

\item \textbf{Implement Closed-Loop Execution}

Forward entropy-derived chunk size 
into the actual runtime kernel.

Report:

\begin{itemize}
\item end-to-end latency
\item runtime overhead
\item GPU utilization
\end{itemize}

This is the single most important requirement.

\item \textbf{Expand Benchmark Diversity}

Include:

\begin{itemize}
\item heterogeneous prompts
\item code-heavy workloads
\item repetitive structured text
\item long-context tasks
\end{itemize}

Demonstrate dynamic switching behavior.

\item \textbf{Add Strong Baselines}

Include:

\begin{itemize}
\item Mamba-2
\item SSD variants
\item FlashAttention-based Transformers
\end{itemize}

\item \textbf{Reconcile Theory with Empirical Behavior}

Provide explanation for:

\begin{center}
entropy contraction under Hadamard
\end{center}

or restrict theoretical claims accordingly.

\item \textbf{Stabilize Threshold Design}

Recalibrate entropy thresholds 
(e.g., $H_{ref} = \log K$) 
to reduce sensitivity to histogram resolution.

\end{enumerate}

\section*{8. Justification of Score}

The core idea—using entropy 
as a runtime scheduling signal—is interesting 
and potentially impactful.

However, the current implementation:

\begin{itemize}
\item lacks closed-loop execution
\item lacks strong competitive baselines
\item shows theoretical inconsistencies
\item demonstrates limited dynamic behavior
\end{itemize}

These collectively prevent the work 
from reaching the maturity level 
typically required for acceptance at NeurIPS.

With substantial system integration 
and expanded evaluation, 
this work could become competitive in future submissions.
