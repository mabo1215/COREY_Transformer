\section*{Review of ``COREY: A Prototype Study of Entropy-Guided Kernel-Level Scheduling for Mamba-1.x SSMs''}

\subsection*{1. Summary}

This paper proposes COREY, a prototype framework for entropy-guided kernel-level scheduling in Mamba-1.x selective state space models (SSMs). 
The central idea is to use activation entropy as a runtime signal to select chunk sizes and scheduling boundaries, aiming to reduce memory traffic and improve scheduling efficiency in GPU execution.

The authors present a two-tier validation framework:

\begin{itemize}
\item Tier-1: A Python-based deterministic cost model demonstrating surrogate latency and DRAM traffic reduction.
\item Tier-2: Real-checkpoint feasibility experiments using Mamba-370M and Mamba-1.4B on RTX 3070 and RTX 3090 GPUs, validating the runtime entropy hook.
\end{itemize}

The work also explores theoretical aspects of entropy-guided scheduling and discusses Hadamard reparameterization as a possible future extension for low-bit inference, although this extension is not experimentally validated.

Overall, the paper positions itself as a systems-oriented prototype study focusing on runtime scheduling rather than full deployment-level acceleration.

\subsection*{2. Strengths}

\begin{itemize}

\item \textbf{Clear Problem Motivation}

The paper correctly identifies memory-bandwidth limitations in selective SSM pipelines as a major performance bottleneck and positions scheduling as an important system-level optimization opportunity.

\item \textbf{Well-Scoped Contribution Claims}

The authors explicitly limit their claims to prototype-level validation and avoid overstating deployment performance. This transparency is commendable and improves credibility.

\item \textbf{Novel Runtime Signal Design}

Using activation entropy as a runtime scheduling signal is conceptually interesting and potentially generalizable to other dynamic workload scenarios.

\item \textbf{Multi-Tier Validation Strategy}

Separating synthetic prototype evaluation from real-checkpoint feasibility testing improves methodological clarity and reduces risk of overgeneralization.

\item \textbf{Strong Theoretical Structure}

The paper provides formal arguments relating entropy properties to fusion feasibility and variance behavior, demonstrating mathematical maturity.

\item \textbf{Implementation Awareness}

The work demonstrates practical understanding of GPU kernel execution constraints such as shared-memory limitations and launch overhead.

\end{itemize}

\subsection*{3. Weaknesses}

\begin{itemize}

\item \textbf{Limited End-to-End Performance Validation}

The current implementation does not integrate entropy-based chunk selection into the actual inference path. 
The entropy hook remains passive, meaning real deployment-level speedup is not yet demonstrated.

This substantially weakens the practical significance of the results.

\item \textbf{Weak Real-World Adaptivity Evidence}

The entropy distribution across real prompts appears tightly clustered, leading to identical chunk-size selection across all prompts. 
This suggests limited real-world adaptivity benefits.

\item \textbf{Synthetic-to-Real Gap}

Much of the theoretical justification relies on synthetic heavy-tailed distributions that do not match real checkpoint activation distributions.

This weakens the general applicability of the theoretical framework.

\item \textbf{Incomplete Baseline Comparisons}

The evaluation lacks comparisons against:

\begin{itemize}
\item Transformer-based fused attention methods
\item Mamba-2 or other modern SSM architectures
\item Compiler-level fusion tools
\end{itemize}

Without such baselines, it is difficult to assess competitive advantage.

\item \textbf{Prototype Dependency}

Key results rely heavily on surrogate cost models rather than fully hardware-measured pipelines.

\item \textbf{Potential Hyperparameter Sensitivity}

Threshold-based entropy discretization may require significant per-model tuning.

\end{itemize}

\subsection*{4. Technical Correctness}

Overall, the technical content appears sound.

\begin{itemize}

\item The entropy estimation method is standard and correctly defined.
\item The scheduling formulation is consistent with kernel grouping logic.
\item The memory-bound modeling assumptions are realistic.
\item The theoretical claims are properly scoped and conditionally stated.
\end{itemize}

However:

\begin{itemize}

\item The applicability of Theorem 1 is restricted to synthetic regimes.
\item Some theoretical insights appear descriptive rather than predictive.

\end{itemize}

No obvious mathematical errors were detected.

\subsection*{5. Experimental Evaluation}

\subsubsection*{Strengths}

\begin{itemize}

\item Uses real GPU hardware (RTX 3070 and RTX 3090)
\item Includes latency benchmarking
\item Demonstrates reproducible synthetic workloads
\item Provides ablation studies on entropy thresholds

\end{itemize}

\subsubsection*{Weaknesses}

\begin{itemize}

\item No full inference pipeline replacement
\item Limited dataset coverage
\item Missing large-scale deployment benchmarks
\item Lack of energy or memory-bandwidth measurements
\item No realistic heterogeneous workload experiments

\end{itemize}

\subsection*{8. Novelty and Significance}

\textbf{Novelty: Moderate}

The entropy-guided scheduling concept is interesting but incremental relative to:

\begin{itemize}

\item Adaptive tiling
\item Dynamic fusion scheduling
\item Runtime resource-aware execution

\end{itemize}

\textbf{Significance: Moderate}

Potential impact exists if:

\begin{itemize}

\item Full integration demonstrates real speedups
\item Cross-model generalization is proven
\item Hardware-level gains scale to production workloads

\end{itemize}

Currently, significance is limited by prototype status.

\subsection*{9. Broader Impact Evaluation}

The broader impact section is thoughtful and responsible.

Potential positive impacts:

\begin{itemize}

\item Reduced energy usage
\item Improved long-context inference efficiency
\item Lower infrastructure cost
\end{itemize}

Potential risks:

\begin{itemize}

\item Enables larger-scale inference deployment
\item May accelerate automated content generation pipelines
\end{itemize}

Ethical framing is appropriate.

\subsection*{10. Questions for the Authors}

\begin{enumerate}

\item Why does real activation entropy collapse into a narrow range across prompts?

\item Can entropy-guided scheduling outperform static tuning under real-world heterogeneous workloads?

\item What is the estimated overhead when entropy-driven chunk selection becomes active rather than passive?

\item How sensitive is performance to entropy bin count?

\item How does this method compare against compiler-based automatic fusion frameworks?

\end{enumerate}

\subsection*{11. Suggestions for Improvement}

\begin{enumerate}

\item Implement full runtime chunk selection integration.

\item Add end-to-end latency benchmarks.

\item Include comparisons with:

\begin{itemize}
\item FlashAttention-style fusion
\item Mamba-2
\item Transformer baselines
\end{itemize}

\item Evaluate heterogeneous workloads.

\item Provide memory bandwidth profiling.

\item Extend evaluation to multi-node distributed environments.

\item Simplify theoretical sections.

\item Add visualization of entropy distribution dynamics.

\end{enumerate}

\subsection*{12. Limitations Assessment}

The paper appropriately acknowledges major limitations, including:

\begin{itemize}

\item Passive runtime hook
\item Limited prompt diversity
\item Absence of full kernel fusion
\item Restricted hardware coverage

\end{itemize}

This transparency is commendable.

\subsection*{13. Overall Recommendation}

\textbf{Rating: Borderline Reject (Weak Reject)}

Reasoning:

\begin{itemize}

\item Conceptually interesting
\item Well-written
\item Methodologically careful
\item But lacking full end-to-end validation
\item Prototype-level results insufficient for top-tier acceptance
\end{itemize}

If full integration were demonstrated, the recommendation could shift significantly.
