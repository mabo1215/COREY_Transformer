\section*{4. Major Issues}

\subsection*{J1. The paper still does not close the main systems loop on real checkpoints}

The strongest remaining weakness is that the inline active scheduler in Section 6.3 computes entropy and selects a chunk, but the selected chunk is \emph{not yet passed into the real checkpoint scan execution path}. As written, the real-checkpoint contribution is an inline monitoring-and-decision mechanism with measured overhead, not a closed-loop scheduler that demonstrates end-to-end serving benefit. The direct speedup evidence remains the synthetic-input kernel benchmark in Section 6.2.

\subsection*{J2. The current real-workload evidence weakens the case for runtime adaptivity}

The manuscript itself reports that across 80 real LongBench prompts all measured prompts map to the same coarse chunk bucket, while the W1 benchmark shows that a one-time static oracle (\texttt{Static-512}) is 35\% faster than COREY on the showcased regime. This means the current evidence supports COREY primarily as an \emph{automatic static tuner within one workload regime}, not as a runtime policy whose adaptivity is necessary on heterogeneous workloads.

\subsection*{J3. The manuscript knowingly evaluates a miscalibrated default}

Section 6.2 now recommends the principled choice $H_{\text{ref}}=\log K$, and Appendix A.16--A.17 shows that the current default $H_{\text{ref}}=8.0$ is systematically conservative for $K=256$. This is not a cosmetic issue: by the paper's own appendix, the calibration choice contributes directly to the gap between COREY and the static oracle. The current version therefore evaluates a setting that the manuscript itself argues is not the right one.

\subsection*{J4. The Hadamard / quantization thread occupies too much of the main body relative to its validated role}

The paper does a careful job of disclaiming the Hadamard extension, but that is also the problem: the extension is not part of the validated runtime contribution, and the key theorem is explicitly shown not to transfer to real checkpoint activations. For a page-limited NeurIPS paper, the current main-body allocation is too generous to a side story whose empirical assumptions fail on the real system.

\subsection*{J5. Evaluation breadth is still below the bar for a main-track systems paper}

The checkpoint story still relies on four LongBench tasks with 20 samples each, one external baseline scale used only as an architectural sanity check, a partially pending Mamba-2.8B table, and some appendix entries that are suppressed or estimated due to instability. This is enough for a feasibility note, but not enough for a persuasive main-track systems contribution.

\section*{5. Minor Issues}

\subsection*{M1. The standalone appendix artifact is not self-consistent}

The local \texttt{paper/appendix.pdf} build still produces unresolved references and an empty bibliography warning. While NeurIPS reviews the single submission PDF, this separate artifact is still part of the repository's Stage 9 package and should be made self-contained.

\subsection*{M2. Reporting mixes measured, estimated, and extrapolated quantities too often}

The paper is generally honest about this, but the reader still has to keep track of which numbers are kernel-timed, checkpoint-timed, estimated from a static sweep, or extrapolated from layer sampling. This creates avoidable cognitive load.

\subsection*{M3. The submission should lean more explicitly into the ``Concept \& Feasibility'' contribution type}

The present manuscript is strongest when read as a concept-and-feasibility paper. That positioning should be made even more explicit in the front matter and submission metadata.

\subsection*{M4. Too many main-text claims require appendix lookups to become properly qualified}

The main paper repeatedly asks the reader to look up critical caveats in the appendix. That is usually a sign that the main story is overpacked for the available page budget.

\section*{6. Consolidated Recommendation}

I would not accept the paper in its current form for NeurIPS 2026. However, I do think there is a publishable idea here if the manuscript is reduced to a sharper and more defensible contribution:

\begin{itemize}[leftmargin=1.5em]
\item a venue-compliant, 9-page concept-and-feasibility paper;
\item one clearly validated story about runtime entropy as a control signal for Mamba chunk selection;
\item a direct comparison against matched static alternatives on the same real checkpoint path;
\item a much smaller role for the prospective Hadamard extension in the main paper.
\end{itemize}

\subsection*{S1. Reduce the main body to nine content pages and refocus the narrative}

Compress the current main body from 11 content pages to 9. The most direct route is to move most of the Hadamard/quantization material, secondary diagnostic commentary, and any non-essential proof discussion out of the main text and into the appendix. The main paper should concentrate on one thesis: runtime entropy as a scheduler signal for Mamba-1.x chunk selection.

% RULES: agents/pipeline/academic-paper.md
\subsection*{S2. Close the checkpoint-side loop or narrow the claim further}

The highest-value revision is to route the selected chunk into the real checkpoint scan path and report the resulting end-to-end latency under the same serving setup used in Section 6.3. If that integration is still infeasible, then the paper must narrow its claim even more sharply and present the current contribution as an inline control-path feasibility result plus a separate kernel-level performance study.

% RULES: agents/pipeline/academic-paper.md
\subsection*{S3. Re-evaluate the runtime policy against matched static baselines under the principled calibration}

The paper should report checkpoint-path comparisons against matched static baselines (\texttt{static-64}, \texttt{static-256}, and preferably \texttt{static-512}) on the same real workload. In addition, the main evaluation should no longer rely on the known-miscalibrated default $H_{\text{ref}}=8.0$ if the paper's own appendix shows that $H_{\text{ref}}=\log K$ is the principled setting.

\subsection*{S4. Demonstrate when runtime adaptation is actually needed}

Either add a heterogeneous real workload mixture in which prompts span multiple entropy regimes and therefore trigger meaningfully different chunk choices, or rewrite the paper so that COREY is explicitly described as an automatic one-regime tuner rather than a broadly adaptive online scheduler.
